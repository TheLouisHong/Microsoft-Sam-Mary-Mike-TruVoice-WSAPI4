Build Environment

To build speech-aware applications, you must have:

  • Microsoft Visual C++ 4.0 or later.
  • Windows 95/98 or Windows NT 4.0 (or later)
  • Put $(SPEECHDK)\Include in your path. $(SPEECHDK) is usually c:\program files\Microsoft Speech SDK. Don't forget to put the directory into Directories tab in Tools/Options in the Microsoft Development Studio.
  • To build the samples put $(SPEECHDK)\samples\include in your path. Don't forget to put the directory into Directories tab in Tools/Options in the Microsoft Development Studio.
  • Put $(SPEECHDK)\Lib in your library path. $(SPEECHDK) is usually c:\program files\Microsoft Speech SDK. Don't forget to put the directory into Directories tab in Tools/Options in the Microsoft Development Studio.

    If you build a debug version of a sample that links to SpchWrap.lib, the linker may produce a warning: "LINK : warning LNK4098: defaultlib "MSVCRTD" conflicts with use of other libs; use /NODEFAULTLIB:library". You can ignore this warning, if you wish, or you can build a debug version of SpchWrap.lib and copy it into your library directory.

  • Note: Some of the samples also require that the following 2 win32sdk directories be included: \mstools\include and \mstools\include\win95
  • We recommend you build from the Microsoft Development Studio.

Building Samples Under MSVC 4.x

Some of the samples in the SD have Visual C++ 5.0 makefiles (and .dsp files). If you open these under Visual C++ 4.x, the makefiles will not work right away. To get the makefiles to work under Visual C++ 4.x:

  • Open the .mak file as a workspace.
  • Say yes to all the questions.
  • Edit all occurrences of "::" to ":" in the .mak file remove of the occurrences of capitalized "/FD" command line parameters.
  • Now the compilation results will probably be placed in a subdirectory other than WinDebug or WinRel. Change your build/settings/executable option to point at the subdir\theprogram.exe.
  • In addition, if you have any directory location errors in your development environment tools\options\directories tab, the dev environment probably would have still worked with other 4.2 compatible mak files -- but will fail now. Review all the directory options, "executable", "include", "library", etc, review all drive and directory specifications listed therein, and make sure they are correct and current.
